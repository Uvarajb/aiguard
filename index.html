Private AI Gateway — LiteLLM + Zscaler AI Guard

✅ Live Version 1.0 June 2026 litellm zscaler ai-guard llm docker ollama security guardrails

1. Overview

This page documents the design, deployment, and operation of a Private AI Gateway that combines LiteLLM (an OpenAI-compatible LLM proxy) with Zscaler AI Guard to provide enterprise-grade AI security guardrails for self-hosted Large Language Models.

Purpose: Enable teams to run private LLMs (via Ollama) while enforcing Zscaler security policies on every AI interaction — blocking PII leakage, prompt injection, toxicity, and data exfiltration before they reach or leave the model.

  • Full audit trail via PostgreSQL
  • Admin UI at http://localhost:4000/ui

2.3 Data Flow

🔄
End-to-End Request Pipeline
  1. Client sends POST /v1/chat/completions with Bearer token to LiteLLM :4000
  2. LiteLLM validates virtual key against PostgreSQL
  3. LiteLLM logs request metadata (tokens, model, user) to PostgreSQL
  4. If guardrail mode = during_call: prompt is sent to Zscaler AI Guard BEFORE model
  5. Zscaler AI Guard evaluates prompt against Policy 232314 (PII, Injection, Toxicity, Jailbreak)
  6. If BLOCK: LiteLLM returns HTTP 400 {"error": "Guardrail blocked this request"} — model never called
  7. If ALLOW: LiteLLM forwards prompt to Ollama :11434
  8. Ollama generates response using llama3 model
  9. If guardrail mode = post_call: response sent to Zscaler AI Guard for output scan
  10. If ALLOW: response returned to client
  11. Spend / token usage updated in PostgreSQL

5. Deployment

5.1 Step-by-Step Deployment

  1. Clone or create your project directory and add all configuration files (docker-compose.yml, litellm_config.yaml)
  2. Create the .env file with your credentials (see Section 4.3)
  3. Pull and start all services:
    bash 
    docker compose up -d
  4. Pull the llama3 model into Ollama:
    bash 
    docker exec ollama ollama pull llama3
  5. Verify all containers are running:
    bash 
    docker compose ps
  6. Test the endpoint (see Section 6)
  7. Access admin UI at http://localhost:4000/ui

7. Zscaler AI Guard Policies

7.1 Policy Overview

Policy IDNameDetectors Enabled
232314Primary AI Guard PolicyPII, Prompt Injection, Toxicity, Jailbreak
172823Output Scan PolicyPII (output), Sensitive Data
123445Strict Mode PolicyAll detectors + custom keywords

7.2 Guardrail Modes

ModeWhen Scan OccursModel Called if Blocked?Use Case
during_callBefore prompt sent to model❌ NoBlock bad input early
post_callAfter model response✅ Yes (but response blocked)Scan output for PII/data leakage
pre_callBefore any processing❌ NoEarliest gate — auth checks

7.3 Policy Precedence

ℹ️
Guardrail Evaluation Order Guardrail evaluation order: pre_call → during_call → [LLM generates response] → post_call. A BLOCK at any stage returns HTTP 400 immediately and stops the pipeline. The model is never called if a pre_call or during_call guardrail triggers a BLOCK.

11. Production Hardening

11.1 Security Checklist

  • Use HTTPS/TLS termination (nginx reverse proxy or Traefik) in front of LiteLLM
  • Store secrets in AWS Secrets Manager or HashiCorp Vault — not in .env files
  • Set strong, random LITELLM_MASTER_KEY (minimum 32 characters)
  • Restrict PostgreSQL to Docker internal network only (remove port 5432 from host binding)
  • Restrict Ollama to Docker internal network only (remove port 11434 from host binding)
  • Enable virtual key rotation policy
  • Set per-key and per-team budget limits to prevent runaway usage
  • Enable audit logging to external SIEM (Splunk, Sentinel, etc.)
  • Monitor guardrail block rates — a spike is an attack indicator
  • Regularly update LiteLLM Docker image for security patches

11.2 Scaling Considerations

  • Run multiple LiteLLM instances behind a load balancer (nginx, AWS ALB, Traefik)
  • Use managed PostgreSQL (AWS RDS, Google Cloud SQL) for production database
  • Use Ollama with GPU support (ollama/ollama:latest-gpu) for faster inference
  • Consider adding more models — mistral, codellama — for different use cases
  • Use the Zscaler regional API endpoint closest to your deployment for lowest latency
